# FileInNOut 사용자 플로우

이 문서는 기능 검증과 E2E 테스트 설계의 기준이 되는 주요 사용자 흐름을 정리합니다.

## 로그인과 세션

1. 사용자가 이메일과 비밀번호로 로그인한다.
2. 백엔드는 access token과 refresh cookie를 발급한다.
3. 프론트엔드는 새로고침 뒤 `/auth/reissue`로 세션을 복구한다.
4. 로그아웃 시 refresh cookie와 서버 저장 refresh session을 함께 정리한다.
5. 관리자 전용 모드에서는 일반 회원가입과 OAuth 진입점이 차단되어야 한다.

검증 포인트:

- access token 만료 뒤 API 요청이 refresh/retry 흐름으로 복구되는지 확인한다.
- 로그아웃 뒤 보호 API가 401 또는 403을 반환하는지 확인한다.
- 같은 계정의 웹 세션과 데스크탑 로그인 세션이 서로를 예기치 않게 끊지 않는지 확인한다.
- 보호 라우트 접근 시 로그인 화면으로 이동하는지 Playwright로 확인한다.

## 파일 업로드와 다운로드

1. 사용자가 파일 또는 폴더를 선택한다.
2. 프론트엔드가 업로드 예약 또는 presigned upload 흐름을 시작한다.
3. 업로드 완료 뒤 DB 메타데이터와 object storage 객체가 같은 논리 경로를 가리킨다.
4. 다운로드는 권한 검증 뒤 authenticated endpoint 또는 presigned URL로 처리한다.
5. 휴지통, 이동, 복구, 영구 삭제 뒤 목록과 저장소 사용량이 일치해야 한다.

검증 포인트:

- 대용량/다중 파일 업로드 중 일부 실패가 전체 상태를 깨지 않는지 확인한다.
- DB 저장 실패 또는 object storage 실패 뒤 orphan object 정리 또는 보상 작업이 동작해야 한다.
- 파일 잠금 상태에서는 수정, 삭제, 이동이 제한되어야 한다.

## 공유 폴더

1. 소유자가 파일 또는 폴더를 사용자나 그룹에 공유한다.
2. 공유 권한은 `READ`, `DOWNLOAD`, `UPLOAD`, `WRITE` 중 하나로 계산한다.
3. 폴더 공유는 현재 하위 항목과 이후 생성되는 하위 항목에 상속된다.
4. 수신자는 공유 목록에서 접근하고, 권한에 따라 다운로드/업로드/수정 가능 여부가 달라진다.
5. 공유 취소 뒤 수신자의 웹/데스크탑 접근은 차단되어야 한다.

검증 포인트:

- READ 공유에서 데스크탑 로컬 수정이 서버에 반영되지 않아야 한다.
- WRITE 공유에서 수신자가 추가한 파일이 소유자의 cloud folder에 반영되어야 한다.
- 공유 링크 만료, 다운로드 제한, 비밀번호 정책이 UI와 API에서 같이 적용되어야 한다.

## 워크스페이스 문서

1. 사용자가 문서를 생성하거나 기존 문서를 연다.
2. 페이지 속성, 작업, 일정, 댓글, 첨부 파일을 편집한다.
3. Yjs realtime server가 공동 편집 상태를 동기화한다.
4. 변경 이력, 댓글 mention, 알림, 검색에 반영된다.
5. 공유 권한에 따라 읽기 전용 또는 편집 가능 모드가 적용된다.

검증 포인트:

- 실시간 연결 실패와 문서 저장 실패가 사용자에게 명확히 표시되어야 한다.
- mention 후보, 작업 담당자, 백링크 계산은 계정 기준으로 중복 없이 표시되어야 한다.
- 읽기 전용 사용자는 저장, 첨부, 댓글 권한이 제한되어야 한다.

## 관리자 화면

1. 관리자가 로그인한다.
2. 사용자, 저장소, 공유, 결제, 운영 지표를 확인한다.
3. 계정 상태, 용량, 공유 정책 등 운영 변경을 수행한다.
4. 변경 결과는 감사 로그와 UI에 반영된다.

검증 포인트:

- 관리자 API는 `ROLE_ADMIN` 없이 접근할 수 없어야 한다.
- 다운로드, 차단, 용량 변경 같은 고위험 동작은 감사 이벤트를 남겨야 한다.
- 관리자 화면의 CSV/다운로드 기능은 민감 정보를 노출하지 않아야 한다.
- 관리자 로그인 성공 뒤 메인 화면 진입은 Playwright E2E로 고정한다.

## 데스크탑 동기화

1. 사용자가 데스크탑 앱에서 서버 URL과 계정으로 로그인한다.
2. 동기화할 로컬 폴더를 선택한다.
3. 웹의 My Drive와 공유 폴더가 계정 기준으로 같은 파일 상태를 바라본다.
4. 로컬 변경은 권한이 허용되는 경우 서버에 업로드된다.
5. 서버 변경은 로컬 FileInNOut drive에 반영된다.
6. 충돌 시 기존 로컬 파일을 보존하고 conflict copy를 만든다.

검증 포인트:

- 웹에서 공유된 폴더가 데스크탑의 `Shared/<owner-email>` 범위와 일치해야 한다.
- READ/DOWNLOAD 공유는 로컬에서도 쓰기 제한이 적용되어야 한다.
- 재시작, 오프라인, 토큰 만료, 서버 URL 변경 뒤에도 설정과 사용자 데이터가 보존되어야 한다.

## E2E 우선순위

1. 로그인 유지, 로그아웃, 보호 라우트 redirect
2. 파일 업로드, 다운로드, 삭제, 복구
3. 공유 생성, 권한 변경, 공유 취소
4. 관리자 로그인과 관리자 전용 API 접근 제어
5. 워크스페이스 문서 생성, 저장, 읽기 전용 접근
6. 데스크탑 sync smoke, 충돌, 오프라인, 재시작
